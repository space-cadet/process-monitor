# T17: Multi-Device Dashboard — Syncthing-Inspired Architecture

**Status:** 🔄 In Progress (Planning)
**Created:** 2026-06-22
**Priority:** HIGH

## Problem

The current dashboard only monitors the local device. Deepak wants to:
1. Monitor multiple devices (MacBook, Cloudy server, Ember/Android)
2. View all dashboards in a unified view from mobile
3. Have a simple pairing flow (scan QR code → done)
4. Work across networks (not just local WiFi)

## Design: Syncthing-Inspired Discovery

### Discovery Layers (Syncthing Model)

**Layer 1: Local Discovery (mDNS/Broadcast)**
- Devices on same LAN discover each other via multicast
- No configuration needed
- Fastest, most reliable when co-located

**Layer 2: Global Discovery Server (Optional)**
- Lightweight HTTP service on Cloudy (or hosted)
- Devices register their latest endpoints
- Observer queries to find device addresses
- Privacy: only device IDs and addresses, no data

**Layer 3: Relay (NAT Traversal)**
- When both devices are behind NAT, direct connection fails
- Relay server (on Cloudy) bridges the connection
- Dumb pipe — encrypted data, relay can't read it

### QR Code Pairing Flow

1. **Device** shows QR code containing:
   ```json
   {
     "did": "ABC123-DEF456",
     "name": "sage-macbook",
     "v": 1,
     "endpoints": [
       "http://192.168.1.39:3456",
       "http://sage-macbook.local:3456"
     ]
   }
   ```

2. **Observer** scans QR → stores device ID + endpoints in local DB

3. **Observer** polls `/api/metrics` from each known device every 30 seconds

4. **Device list** renders as cards with: name, battery%, top processes, alerts

### Implementation Options (V1 → V2 → V3)

| Version | Approach | Discovery | Complexity |
|---------|----------|-----------|------------|
| **V1** | Manual QR + polling | QR code only, no server | Low |
| **V2** | QR + local mDNS | QR for first pair, mDNS for reconnection | Medium |
| **V3** | Full Syncthing-style | QR + mDNS + global discovery + relay | High |

### API Design (Per Device)

```
GET /api/identity          → { did, name, version, endpoints[] }
GET /api/metrics?since=ts  → { battery, processes, alerts, timestamp }
GET /api/devices           → List of known devices (for device-to-device sync)
```

### Data Flow

```
Observer → GET /api/metrics?since=lastPoll → Device
         ← { battery: 87%, processes: [...], alerts: [...] }

Observer stores in local SQLite/IndexedDB
Observer renders unified view
```

## Technical Decisions

### Device Identity
- Generate persistent UUID on first run
- Stored in `~/.procmon/device-id` (or equivalent)
- Display name configurable (defaults to hostname)

### QR Code Generation
- Use `qrcode` npm package (or `qrcode-generator` for frontend)
- Include: device ID, name, version, endpoints array
- Rendered in dashboard Settings or Overview tab

### Observer Dashboard
- Can be the same dashboard codebase, running in "observer mode"
- Or a separate lightweight page (mobile-optimized)
- Store paired devices in localStorage (V1) or SQLite (V2+)

### Network Considerations
- **Local WiFi**: mDNS/Bonjour `.local` hostnames, local IPs
- **Same network, different subnets**: May need discovery server
- **Different networks**: Requires relay or VPN (Tailscale)
