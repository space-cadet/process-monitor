## Problem
Spike detection (T6) uses simple thresholds. It catches obvious spikes but misses subtle patterns: a process gradually climbing from 5% to 25% CPU over an hour, or battery draining faster than usual for no obvious reason.

## Solution
Statistical anomaly detection on time-series data:

### Per-Process Anomaly Scoring
- Track each process's historical distribution (mean, stddev, percentiles)
- Flag when current value exceeds mean + 2σ (two standard deviations)
- Score: `(current - mean) / stddev` — higher = more anomalous

### Battery Drain Anomaly
- Learn user's typical drain rate at different times of day
- Flag when current drain exceeds typical by >2σ
- Example: "Your Mac usually drains 0.3%/min at 2pm. Now draining 1.2%/min — 4σ above normal."

### Seasonal Patterns
- Day-of-week effects: higher CPU on weekdays (work apps)
- Time-of-day: different baselines for morning vs evening

## Technical Approach

### Algorithm
- Simple: Moving average + standard deviation (online, no training needed)
- Advanced: Isolation Forest or local outlier factor (requires batch processing)
- Start simple, iterate

### Storage
- New `process_baselines` table with per-process stats
- New `anomaly_events` table for flagged anomalies

### Dashboard
- Anomaly score badge on process list
- "Anomalies" panel showing recent statistical outliers
