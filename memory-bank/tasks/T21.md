# T21: DB Size-Based Cleanup Fix

**Status:** ✅ Complete
**Started:** 2026-07-15
**Completed:** 2026-07-15
**Priority:** HIGH

## Problem

The `cleanupOldSamples()` method in `TimeSeriesDB.ts` only accepted a single `retentionDays` parameter. The second parameter (`maxSizeMB`) was being passed by the orchestrator but silently ignored. This meant:

- DB grew to 608MB (108MB over the 500MB limit)
- Cleanup only ran by age (30 days), never by size
- No automatic size-based rotation was actually happening

Additionally, `process_spikes` table has a foreign key (`snapshot_id` → `snapshots.id`) but the cleanup only deleted from `process_samples` before deleting snapshots, causing `SQLITE_CONSTRAINT_FOREIGNKEY` errors.

## Solution

### 1. Signature Change
```typescript
// Before
 cleanupOldSamples(retentionDays: number): void

// After
 cleanupOldSamples(retentionDays: number, maxSizeMB?: number): void
```

### 2. Size-Based Cleanup Algorithm
After time-based cleanup (delete snapshots older than `retentionDays`):

1. If `maxSizeMB` provided and DB size > target:
2. Query oldest 500 snapshots by timestamp
3. Delete child rows first (foreign key order):
   - `process_samples` where `snapshot_id IN (...)`
   - `process_spikes` where `snapshot_id IN (...)`  ← **was missing**
   - `snapshots` where `id IN (...)`
4. Repeat until under limit or no more rows (max 100 iterations)
5. Run `VACUUM` to reclaim disk space

### 3. Foreign Key Fix
Added `process_spikes` deletion to both:
- Time-based cleanup path (before deleting old snapshots)
- Size-based cleanup path (in the batch loop)

### 4. API Update
Dashboard `/api/cleanup` POST now accepts `maxSizeMB`:
```json
{
  "retentionDays": 30,
  "maxSizeMB": 400
}
```

### 5. Monitor Integration
`Monitor.ts` now passes `this.config.retentionSizeMB` (default 400MB) to cleanup.
