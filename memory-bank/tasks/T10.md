# T10: "What Drained My Battery?" Automated Report

*Created: 2026-06-18*
*Status: ⬜ PENDING*
*Priority: HIGH*

## Problem
Users have raw data (spikes, drain events, battery impact) but no synthesized narrative. Activity Monitor shows real-time; our DB shows history, but nobody wants to write SQL to find out why their battery died.

## Solution
A daily/weekly automated report that reads the DB and produces a human-readable summary:

```
📊 Battery Report — June 17, 2026

🔋 Health: 85% → 78% net (started at 95%, ended at 78%)
⚠️  Drain events: 2 (total 12% lost to rapid drain)
🔥 Spikes: 5 (Chrome x2, Docker x1, VS Code x1, kernel_task x1)
🏆 Top culprits:
  1. Chrome — 23% battery impact score, 4.2h active
  2. Docker Desktop — 18% impact, 2 spikes to 89% CPU
  3. VS Code — 8% impact, 1 long spike during indexing

💡 Insights:
  • Chrome had a 45-min sustained drain period at 14:00
  • Docker spiked 3x when you were on battery — consider quitting it
  • No drain detected during sleep (good!)
```

## Technical Approach

### Report Generator
- `src/core/ReportGenerator.ts` — new module
- Queries: drain events, spikes, battery impact scores from last 24h/7d
- Aggregates per-process impact, finds patterns
- Generates markdown/text report

### Delivery
- CLI: `npx tsx src/query.ts --report --since 24h`
- Telegram: Scheduled daily report via cron (configurable)
- Dashboard: New "Reports" tab with historical reports

### Scheduling
- Configurable: daily at 9am, weekly on Monday, or on-demand
- Stored in DB so reports survive restarts
